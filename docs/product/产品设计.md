# Synkord 产品设计

> 版本：v2.0  
> 更新：2026-06

---

## 一、产品定位

**Synkord 是以业务域为单位的契约管理工具。**

让每个人、每个项目、每个 AI 工具，在任何时候都基于同一份业务数据契约工作。

---

## 二、解决的核心问题

在多人、多项目的团队中，业务数据定义会悄悄漂移：

- 没有一个所有人都认可的、机器可读的数据契约作为基准
- AI 工具（Cursor、Codex 等）生成代码时不知道契约，字段名和类型随意
- 某人修改了共享数据结构，依赖方不知道，联调时才发现问题
- 多个项目对同一个实体的理解各自不同（`userId` vs `user_id` vs `uid`）

---

## 三、核心用户

**核心开发者**：负责多个项目，属于某个团队，项目之间共享业务数据契约。

- 不区分前端 / 后端 / 全栈，只区分**负责的业务域**
- 每个人可以**拥有**某些契约包（负责维护）
- 每个人可以**订阅**其他契约包（依赖消费）

---

## 四、核心概念

### 契约包（Contract Pack）

业务域的数据边界声明，机器可读的 JSON 格式。

```json
{
  "pack": "auth-pack",
  "version": "1.2.0",
  "entities": {
    "User": {
      "table": "users",
      "fields": {
        "id":    { "type": "uuid", "primary": true },
        "email": { "type": "string", "maxLength": 255 },
        "status": { "type": "enum", "values": ["active", "suspended"] }
      },
      "relations": {
        "roles": { "type": "many-to-many", "target": "Role", "through": "user_roles" }
      }
    }
  }
}
```

契约包存储在 Git 仓库，天然拥有版本历史和协作审批流程。

### 所有权与订阅

| 关系 | 含义 |
|---|---|
| 拥有（Owner） | 负责维护该契约包，有权发布新版本 |
| 订阅（Consumer） | 项目依赖该契约包，感知版本变化 |

### 业务域

一组内聚的契约包，如：

- `auth-pack`：用户与权限域
- `order-pack`：订单与支付域
- `catalog-pack`：商品与库存域

---

## 五、核心功能

### 5.1 定义契约（Define）

- 用 JSON 格式声明实体、字段类型、关联关系
- 通过 JSON Schema 校验契约文件的合法性
- 版本管理由 Git 负责

### 5.2 分发契约（Distribute）

**给 AI 工具**（MCP Server）

Synkord 运行一个本地 MCP Server，Cursor / Codex / Claude 等工具接入后，AI 在生成代码时自动感知契约：

```
AI："User 实体有哪些字段？"
Synkord："id(uuid), email(string), status(enum: active/suspended)..."
```

**给开发者**（契约看板）

悬浮看板或 IDE 侧边栏，随时查看当前项目依赖的契约详情和版本状态。

### 5.3 感知变化（Notify）

- 契约包发布新版本时，所有订阅项目收到通知
- 通知展示 diff：新增了哪些字段，删除了什么，类型是否变化
- 开发者决定是否升级、如何适配

## 六、产品形态

```
Synkord
├── 桌面 App（Electron，轻量）
│   ├── 契约库管理（浏览、编辑、版本对比）
│   ├── 项目注册表（项目 → 契约订阅关系）
│   ├── 变更通知中心
│   └── 悬浮看板（alwaysOnTop，可贴边）
│
├── MCP Server（后台常驻，localhost）
│   └── 为 Cursor / Codex / Claude 提供契约上下文
│
└── CLI（synkord check）
    └── 在 CI/CD 中校验代码是否符合契约
```

---

## 七、技术架构

| 层 | 技术选型 | 说明 |
|---|---|---|
| 桌面外壳 | Electron + React | 跨平台桌面 App |
| MCP Server | Node.js (TypeScript) | 实现 MCP 协议，常驻后台 |
| 契约存储 | 本地文件系统 + Git | 契约就是 JSON 文件 |
| 契约校验 | ajv（JSON Schema 校验） | 验证契约文件合法性 |
| 本地数据库 | SQLite | 项目注册、订阅关系、通知记录 |
| CLI | Node.js | `synkord check` 命令 |

契约格式定义见 `schemas/contract-v1.json`，样例见 `examples/auth-pack.json`。

---

## 八、MVP 范围

**目标：跑通核心闭环**

1. 能加载本地 Git 目录中的契约包
2. MCP Server 启动，Cursor 接入后 AI 能感知契约
3. 当契约文件变化时，App 能检测并展示通知

**不在 MVP 范围**

- 云端同步 / 团队共享服务
- 代码合规 AST 校验
- VS Code 插件（MCP 优先）
- AI 辅助生成契约
- WebSocket 实时协商

---

## 九、未来方向
