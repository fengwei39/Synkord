## 一、产品定位

### 1.1 一句话定义

**Synkord 是一款桌面端「架构防腐与数据契约协同平台」：面向 3–10 人核心开发团队，提供契约制定、协同发布、AI 辅助开发与合规校验的一体化能力；通过契约库跨项目复用实体标准，强制团队内所有成员及其使用的 AI 工具服从统一数据与接口契约。**

### 1.2 产品本质

Synkord **不是** AI 代码生成器，而是 **架构级数据标准执行器**：

| 维度 | 说明 |
|---|---|
| 模型策略 | 不绑定任何 AI 厂商；支持 GPT、Claude、豆包、通义、Ollama 等云端或本地模型 |
| 约束对象 | 团队所有成员及其使用的 AI 工具，均须服从统一实体与接口契约 |
| 核心目标 | 锁死底层数据标准，防止开发过程中架构意图被逐渐偏离 |
| 复用机制 | 契约库跨项目复用，多仓库共用同一套核心实体 |

**对外话术：**

> Copilot 让 AI 写得更快，Synkord 让团队里的每个人和 AI **不敢写错**。

### 1.3 核心壁垒

| 壁垒 | 说明 |
|---|---|
| 合规约束 | 市面工具卷「生成」，Synkord 卷「合规」——不符合契约的代码过不了流水线 |
| 契约库复用 | 多项目引用同一实体包，避免每个仓库重定义 `User` |
| 模型兼容 | 多模型异构接入；Key 本机 AES 加密，不上云 |
| 桌面离线 | Electron + Rust + SQLite，云端仅做协同同步 |
| 工程闭环 | 定契约 → App/CLI/CI 校验 → 产出始终对齐同一标准 |

### 1.4 产品边界

| 做 | 不做 |
|---|---|
| 约束数据层（实体、字段、类型、关联、命名） | 约束业务逻辑正确性 |
| 约束接口层（API 路径、参数、响应格式） | 绑定特定 IDE 或编辑器 |
| 契约变更管理与合规校验 | 做重度云端 IDE 替代品 |
| | 做通用 DevOps 平台 |

### 1.5 设计原则

1. **契约为唯一真相源** — Single Source of Truth
2. **契约库跨项目复用**，项目只做引用与扩展
3. **不绑 AI**，平台只做调度与约束
4. **本地与云端能力对称** — 契约库、引用、校验逻辑一致
5. **桌面优先、离线可用**
6. **工程闭环内置** — App 为主战场，CLI/CI 嵌入日常流程
7. **校验分层、降级不死锁** — 硬规则 → 智能裁判 → 人工接管
8. **轻量 Agent 辅助** — 主轴仍是架构治理，非自主开发平台

## 二、目标用户与场景

### 2.1 用户模型：核心开发

Synkord 只有一层用户——**核心开发**。团队内所有成员共同制定契约、协同开发、接受校验，不区分角色层级。

| 维度 | 说明 |
|---|---|
| 画像 | 3–10 人精英小团队：Tech Lead、Senior 工程师、全栈开发者 |
| 典型场景 | 创业公司核心技术组、精悍外包小队、工作室研发组 |
| 核心诉求 | 定标准、防腐化、多项目复用契约、多 AI 协同不分裂 |
| 使用方式 | 桌面 App（主）+ CLI/CI（嵌入工程流程） |

### 2.2 工作模式

核心开发的两种工作模式，**不是两种用户等级**：

| 维度 | 本地个人 | 云端团队 |
|---|---|---|
| 场景 | 独立开发者，一人多项目 | 3–10 人协作 |
| 工作区 | 个人工作区 | 团队工作区 |
| 契约库 | 个人契约库 | 团队契约库 |
| 协作 | 无 | 成员、权限、WebSocket、表决 |
| AI Key | 本机加密 | 本机加密，不上云 |
| 存储 | SQLite（唯一真相源） | MySQL + 本地 SQLite 缓存 |
| 离线 | 完全可用 | 可离线编辑，重连后同步 |
| 定价 | 免费试用 | 按团队席位收费 |

**升级路径：**

```
个人本地试用 → 个人契约库打磨 → 发布到团队契约库 → 项目迁移至云端 → 接入 CI 卡点
```

**部署拓扑：**

| 拓扑 | 场景 | 组件 |
|---|---|---|
| 纯单机 | 个人试用 | 桌面 App + SQLite |
| 局域网 | 小团队同办公室 | 桌面 App + 局域网 Sync Server |
| 云端 | 远程分布式团队 | 桌面 App + MySQL + WebSocket |

### 2.3 核心痛点

| 痛点 | 表现 | 解法 |
|---|---|---|
| 架构腐化 | 接口乱改、命名不一致、表结构乱扩展 | 契约强约束 + CI 拦截 |
| 跨端内耗 | 每天对齐前后端接口、DB 字段 | 契约库为单一事实来源 |
| 多 AI 分裂 | 每人用不同 AI，字段命名各自为政 | 契约注入 + 统一校验 |
| 变更失控 | 契约改动影响面不透明 | Diff + 表决发布 |

### 2.4 典型一天（云端团队）

```
09:00  打开 Synkord，查看团队契约库 auth-pack 中的 User 实体
09:30  与队友讨论：Order 需新增 refund_status 字段
10:00  JSON 契约编辑器修改，右侧 ER 图实时更新
10:15  提交变更审查，WebSocket 通知全员
10:30  查看 Diff 图谱，一人驳回「字段类型应为 enum」
10:45  修订后全票通过，正式发布 v1.3.0
11:00  WebSocket 推送，全员 AI 上下文自动重载
14:00  成员 A 用 Agent 生成前端 Interface，App 内校验通过
15:00  成员 B 提交 PR，CI 执行 synkord check，一次通过
16:00  成员 C 本地 synkord check 发现遗漏，修复后合入
```

## 三、核心概念模型

### 3.1 层级结构

```
工作区 (Workspace)
├── type: personal | team
│
├── 契约库 (Contract Library)
│     └── 契约包 (Contract Pack) × N
│           ├── 实体 (Entity) + 字段 / 类型 / 约束
│           ├── 关联 (Relation)
│           └── 规范 (Naming, API 约定)
│
└── 项目 (Project) × N
      ├── imports      — 引用的契约包 + 版本
      ├── extensions   — 对共享实体的合法扩展
      └── local_entities — 项目私有实体
```

### 3.2 关键概念

| 概念 | 定义 |
|---|---|
| 工作区 | 契约库与项目的顶层容器 |
| 契约库 | 工作区级共享资产；个人与团队均有，结构相同 |
| 契约包 | 可复用的版本化模块，如 `auth-pack` |
| 项目 | 业务应用、微服务或代码仓库的逻辑单元 |
| 有效契约 | 运行时合并结果，校验器/CI/Agent 的唯一依据 |

```
有效契约 = merge(引用的契约包, 项目扩展, 项目私有实体)
```

### 3.3 共享与私有

| 放入契约库（跨项目共享） | 留在项目（私有） |
|---|---|
| User、Role、Session | 某后台独有的 DashboardWidget |
| Order、Product | 一次性原型实体 |
| 命名规范、ID 类型、时间格式 | 项目特有业务实体 |
| 通用 API 路径约定 | 项目专属接口 |

### 3.4 引用规则

```yaml
project: ecommerce-api
imports:
  - pack: auth-pack
    version: ^1.2.0
extensions:
  - entity: User
    add_fields: [last_login_ip]
local_entities:
  - name: AdminAuditLog
    fields: [...]
```

| 操作 | 规则 |
|---|---|
| 扩展共享实体 | ✅ 允许新增字段、新增关联 |
| 修改共享实体 | ❌ 禁止改已有字段名/类型、删字段 |
| 修改核心定义 | 必须到契约库操作，走版本升级与发布 |

**版本语义：** `minor`（加字段）可选同步；`breaking`（改类型/删字段）需人工确认。

## 四、契约规范

### 4.1 规范选型

**JSON Schema 核心规范 + Prisma 关系映射概念**

| 优势 | 说明 |
|---|---|
| 表达能力 | 精确定义实体、DB 映射、主外键、校验规则 |
| Git 友好 | 纯文本 diff/merge/blame |
| AI 亲和 | LLM 对 JSON Schema 强 Zero-shot 解析 |
| 生态互通 | 可导出 SQL、Prisma、OpenAPI、TS interface |

### 4.2 文件示例

```json
{
  "$schema": "https://synkord.dev/schema/contract-v1.json",
  "pack": "auth-pack",
  "version": "1.2.0",
  "conventions": {
    "id_type": "uuid",
    "naming": { "db": "snake_case", "api": "camelCase", "java": "PascalCase" },
    "timestamps": { "created_at": "datetime", "updated_at": "datetime" }
  },
  "entities": {
    "User": {
      "table": "users",
      "fields": {
        "id":     { "type": "uuid", "primary": true },
        "email":  { "type": "string", "maxLength": 255, "unique": true },
        "status": { "type": "enum", "values": ["active", "suspended"] }
      },
      "relations": {
        "roles": { "type": "many-to-many", "target": "Role", "through": "user_roles" }
      }
    }
  }
}
```

### 4.3 多端命名映射

```
契约字段 email
  → DB:        email     (snake_case)
  → API/TS:    email     (camelCase)
  → Java DTO:  email     (camelCase)
  → Go:        Email     (PascalCase)
```

## 五、校验引擎

### 5.1 三层架构

```
代码产出 / PR 提交
    ↓
 L1 极速过滤（Rust，毫秒）     ← CI 默认
    ↓  --deep
 L2 LLM 智能裁判（可选）       ← 重要 PR / 契约审阅
    ↓  企业版
 L3 AST 插件（开放接口）       ← 企业定制
```

| 层级 | 技术 | 示例 | 场景 |
|---|---|---|---|
| L1 | Rust 正则/JSONPath | 缺 `email` 字段；DB 列名 camelCase | 每次 PR |
| L2 | LLM Judge | DTO 未完整表达 Order 必填字段 | `synkord check --deep` |
| L3 | AST 插件 | 企业 Java 注解规范 | 企业版 |

### 5.2 校验覆盖

| 产出物 | P0 | P1 |
|---|---|---|
| SQL 建表 | ✅ | ✅ |
| TS Interface | ✅ | ✅ |
| Java Entity/DTO | — | ✅ |
| Go struct | — | ✅ |
| Vue/React API | — | ✅ |
| OpenAPI | — | ✅ |

## 六、协同与降级

### 6.1 契约变更表决（云端团队）

> 本地个人模式无此流程，直接发布即可。

```
编辑待提交 → 提交待表决 → 待发布 → 已发布
     ↑            │
     └── 驳回 ────┘
```

| 阶段 | 动作 |
|---|---|
| 编辑待提交 | WebSocket 广播编辑锁，其他人只读 |
| 提交待表决 | 生成 Diff 图谱推送全员 |
| 待发布 | 表决通过，自动格式校验 |
| 已发布 | 新版入库，推送全员，AI 上下文重载 |
| 驳回 | 退回编辑，附理由 |

**表决规则：** 默认全票通过；可切换多数决（2/3）；Owner 可紧急发布（记审计日志）。

### 6.2 降级机制

| 级别 | 名称 | 动作 |
|---|---|---|
| Level 1 | 机器自愈 | 违规报告 → 打回 AI 重写，最多 3 次 |
| Level 2 | 局部采纳 | Diff 高亮 → 手动修改标红区域 → 合规后保存 |
| Level 3 | 人工接管 | 冻结 Agent →「强制采纳」或「废弃」（记审计日志） |

## 七、Agent 与多模型

### 7.1 Agent 定位

Agent 是辅助能力，不是产品主轴：

| 场景 | 说明 |
|---|---|
| AI 起草契约 | 从需求/源码反向生成 JSON Schema |
| AI 修复违规 | Level 1 自动重试的执行者 |
| AI 辅助评审 | 表决阶段 L2 Judge 辅助评估变更影响 |

### 7.2 多模型接入

| 扩展线 | 职责 | 示例 |
|---|---|---|
| Model Provider | 发 AI 请求 | OpenAI、Claude、豆包、Ollama、自定义 URL |
| Capability/Tool | Agent 可用能力 | 文件、终端、Git、契约读写、MCP、Skills |

- Key 本机 AES 加密，平台不托管
- 请求前注入有效契约 + Skills
- 同项目不同成员可用不同模型

### 7.3 Skills

| 类型 | 范围 | 内容 |
|---|---|---|
| 个人 Skills | 本地工作区 | 编码偏好 |
| 团队 Skills | 云端工作区 | 代码规范、技术栈模板 |

Skills 与契约形成双重约束：数据结构 + 编码风格。

## 八、产品界面与工程闭环

### 8.1 桌面 App

```
Synkord 桌面 App
├── 工作区切换（个人本地 / 云端团队）
├── 契约库（契约包管理、版本历史）
├── 项目列表（引用关系、有效契约预览）
├── 双栏架构工作台
│   ├── 左：JSON/YAML 编辑器（Monaco）
│   └── 右：实体拓扑图 / ER 图
├── 变更管理（Diff、表决、发布历史）
├── AI 助手（起草/修复）
├── 校验报告（违规列表、行号定位）
└── 设置（AI 模型、Skills、CLI）
```

### 8.2 CLI 与 CI

```bash
# 安装
npm install -g @synkord/cli

# 常用命令
synkord init                              # 项目关联团队契约
synkord check ./src                       # 本地 L1 校验
synkord check --deep ./src                # L1 + L2 深度校验
synkord report                            # 查看当前契约版本
```

**GitHub Actions：**

```yaml
name: Synkord Contract Check
on: [pull_request]
jobs:
  contract-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: synkord/check-action@v1
        with:
          contract-url: ${{ secrets.SYNKORD_CONTRACT_URL }}
          level: basic
          fail-on-error: true
```

### 8.3 团队工程闭环

```
团队在 Synkord 定契约并发布
  → 成员在 App 内开发，AI 产出自动校验
    → 代码库接入 synkord check CI
      → 任何产出不合规即标红
        → 团队付费扩展云端协同
```

## 九、技术架构

### 9.1 技术栈

| 层 | 选型 | 职责 |
|---|---|---|
| 壳 | Electron + Node.js | 跨平台桌面、IPC |
| UI | TS + React + Radix + Monaco | 双栏工作台、Diff |
| 内核 | Rust (sidecar) | 校验、契约解析、模型调度、导出 |
| 本地存储 | SQLite | 个人全量；团队本地缓存 |
| 云端存储 | MySQL + 对象存储 | 团队、契约库、版本 |
| 协同 | WebSocket + REST | 编辑锁、表决、推送 |

### 9.2 进程架构

```
React 渲染进程
  ↓ Electron IPC
Node.js 主进程
  ↓ IPC / stdio
Rust 内核
  ├── 契约解析器
  ├── L1 校验引擎 / L2 Judge 调度器
  ├── Model Provider 适配器
  ├── 导出器
  └── 沙箱执行器
```

### 9.3 存储职责

| 数据 | 本地个人 | 云端团队 |
|---|---|---|
| AI Key | 本机加密 | 同左 |
| 契约库 | SQLite | MySQL + SQLite 缓存 |
| 审计日志 | 本地 | 云端（付费） |
| 离线真相源 | 本地 SQLite | 重连后以云端为准 |

## 十一、MVP 分期

### Phase 1 — P0（8–12 周）

**目标：跑通「定契约 → 校验 → CI 拦截」闭环。**

| 模块 | 范围 |
|---|---|
| 桌面 App | Electron + React + Monaco 双栏工作台 |
| 工作区 | 个人工作区 + 个人契约库 + 多项目引用 |
| 契约 | JSON Schema 子集 + ER 实时预览 |
| 导出 | MySQL SQL + TS Interface |
| 校验 | L1 Rust 引擎（TS + SQL）；L2 `--deep` |
| 模型 | OpenAI + Ollama |
| Agent | 轻量：AI 起草契约、自动修复 |
| 工程 | CLI + GitHub Actions 模板 |
| 存储 | SQLite |
| 发布 | 手动发布（无表决） |

**验证标准：** 两项目共用 `auth-pack`；TS 代码缺 `User.email` → App 标红 → CLI 复现 → CI 拦截。

### Phase 2 — 团队协同（+8 周）

云端团队工作区、成员权限、WebSocket 表决、版本管理、个人→团队迁移、局域网部署。

### Phase 3 — 全栈增强（+8 周）

Java/Go/Vue/React 校验导出、团队 Skills、Level 2 局部采纳、VS Code 插件、审计日志。

### Phase 4 — 企业版

L3 AST 插件、可配置表决、Automations、多团队管理、私有化部署。

## 十二、商业化与竞品

### 12.1 增长漏斗

```
个人免费试用 → 邀请队友组团队 → 接入 CI → 团队付费
```

## 附录：术语表

| 术语 | 英文 | 定义 |
|---|---|---|
| 核心开发 | Core Developer | 唯一用户类型，共同定契约、开发、校验 |
| 工作区 | Workspace | 个人或团队的顶层容器 |
| 契约库 | Contract Library | 工作区内可复用契约包的集合 |
| 契约包 | Contract Pack | 实体与规范的版本化模块 |
| 有效契约 | Resolved Contract | 契约包 + 扩展 + 私有实体合并后的运行时标准 |
| 架构防腐 | Anti-corruption | 防止开发过程中架构意图被逐渐偏离 |
| 校验探针 | Contract Probe | CLI/CI 中的契约合规检查 |
