## 一、产品概述

### 1.1 一句话定义

**Synkord 是一款桌面端「架构防腐与数据契约协同平台」：面向 3–10 人核心开发团队，提供契约制定、协同发布、AI 辅助开发与合规校验的一体化能力；通过契约库跨项目复用实体标准，强制团队内所有成员及其使用的 AI 工具服从统一数据与接口契约。**

### 1.2 核心设计目标

Synkord **不是** AI 代码生成器，而是 **架构级数据标准执行器**：

- **不绑定**任何 AI 厂商；用户自由接入 GPT、Claude、豆包、通义、Ollama 等云端或本地模型
- **所有 AI 与人类开发者**必须服从平台定义的统一实体与接口契约
- **锁死底层数据标准**，防止团队成员及各类 AI 助手在开发过程中擅自篡改架构意图
- **契约库跨项目复用**，多微服务、多前端、多代码仓库共用同一套核心实体

### 1.3 核心差异化壁垒

| 壁垒 | 说明 |
|---|---|
| **合规约束** | 市面工具卷「生成」，Synkord 卷「合规」——不符合契约的代码过不了流水线 |
| **契约库跨项目复用** | 组织/个人契约库存放可复用实体包，多项目引用，避免每个仓库重定义 `User` |
| **极致模型兼容与隐私** | 多模型异构接入；个人 Key 与本地模型地址本机 AES 加密，不上云 |
| **纯桌面离线优先** | Electron + Rust 内核 + SQLite，极客级响应；云端仅做协同同步 |
| **CI 工程闭环** | 团队定契约 → 本地/CI 自动校验 → 成员与 AI 产出始终对齐同一标准 |

### 1.4 产品边界

**做什么：**

- 约束数据层结构（实体、字段、类型、关联、命名）
- 约束接口层结构（API 路径、参数、响应格式）
- 契约变更管理与合规校验

**不做什么：**

- 不约束业务逻辑正确性
- 不绑定特定 IDE 或编辑器
- 不做重度云端 IDE 替代品
- 不做通用 DevOps 平台

**对外话术：**

> Copilot 让 AI 写得更快，Synkord 让团队里的每个人和 AI **不敢写错**。

## 二、设计原则

1. **契约为唯一真相源（Single Source of Truth）**
2. **契约库跨项目复用，项目只做引用与扩展**
3. **不绑 AI，平台只做调度与约束**
4. **本地与云端能力对称**（契约库、引用、校验逻辑一致；差异在协作与存储）
5. **桌面优先、离线可用**
6. **工程闭环内置**（桌面 App 为主战场，CLI/CI 嵌入核心开发日常流程）
7. **校验分层、降级不死锁**（硬规则 → 智能裁判 → 人工接管）
8. **轻量 Agent 辅助**（AI 起草契约、修复违规；主轴仍是架构治理）

## 三、目标用户

### 3.1 唯一用户：核心开发

Synkord 只有一层用户结构——**核心开发**。不区分「定标准的架构组」与「落地执行的下游研发」，团队内所有成员均为核心开发，共同制定契约、协同开发、接受校验。

| 维度 | 说明 |
|---|---|
| **画像** | 3–10 人精英小团队：Tech Lead、Senior 工程师、全栈开发者 |
| **典型场景** | 创业公司核心技术组、精悍外包小队、工作室研发组 |
| **核心诉求** | 定标准、防腐化、多项目复用契约、多 AI 协同不分裂 |
| **使用方式** | 完整桌面 App（主）+ CLI/CI（嵌入日常工程流程） |

### 3.2 本地个人 vs 云端团队

核心开发在不同规模下的两种工作模式，**不是两种用户等级**：

| 模式 | 场景 | 说明 |
|---|---|---|
| **本地个人** | 独立核心开发者 | 一人多项目，个人契约库，完全离线 |
| **云端团队** | 3–10 人核心开发组 | 组织契约库、协同表决、WebSocket 同步 |

两种模式的用户都是核心开发，能力对称，差异仅在协作规模。

### 3.3 核心痛点

| 痛点 | 表现 | Synkord 解法 |
|---|---|---|
| **架构腐化** | 接口被乱改、驼峰/下划线不一致、表结构胡乱扩展 | 契约强约束 + CI 拦截 |
| **跨端协同内耗** | 每天对齐前后端接口、DB 字段变动 | 契约库为单一事实来源 |
| **多 AI 产出分裂** | 每人用不同 AI，字段命名各自为政 | 契约注入 + 统一校验 |
| **变更不可控** | 契约改动影响面不透明 | 变更 Diff + 表决发布流程 |

### 3.4 典型用户一天

```
09:00  打开 Synkord，查看组织契约库 auth-pack 中的 User 实体
09:30  与 teammate 讨论：Order 实体需新增 refund_status 字段
10:00  在 JSON 契约编辑器中修改，右侧 ER 图实时更新
10:15  提交变更审查，WebSocket 通知团队其他成员
10:30  全员查看 Diff 图谱，一人驳回「字段类型应为 enum」
10:45  修订后重新提交，全票通过，正式发布 v1.3.0
11:00  WebSocket 推送全员，本地 AI 上下文自动重载
14:00  成员 A 用 Agent 生成前端 Interface，App 内校验通过
15:00  成员 B 提交 PR，GitHub Actions 执行 synkord check，一次通过
16:00  成员 C 本地 synkord check 发现遗漏字段，修复后合入
```

## 四、核心概念模型

### 4.1 层级结构

```
工作区 (Workspace)
├── type: personal（本地个人）| organization（云端组织）
│
├── 契约库 (Contract Library)
│     └── 契约包 (Contract Pack) × N
│           ├── 实体 (Entity)
│           ├── 字段 / 类型 / 约束
│           ├── 关联 (Relation)
│           └── 规范 (Naming, API 约定)
│
└── 项目 (Project) × N
      ├── imports: 引用的契约包 + 版本
      ├── extensions: 对共享实体的合法扩展
      └── local_entities: 项目私有实体
```

### 4.2 关键定义

| 概念 | 定义 |
|---|---|
| **工作区** | 契约库与项目的顶层容器 |
| **契约库** | 工作区级共享资产；个人本地与云端组织均有，结构相同 |
| **契约包** | 契约库中可复用的版本化模块，如 `auth-pack`、`commerce-pack` |
| **项目** | 具体业务应用、微服务或代码仓库的逻辑单元 |
| **有效契约** | 运行时合并结果，校验器、CI、Agent 的唯一依据 |

```
有效契约 = merge(引用的契约包, 项目扩展, 项目私有实体)
```

### 4.3 共享与私有边界

| 放入契约库（跨项目共享） | 留在项目（私有） |
|---|---|
| User、Role、Session | 某后台独有的 DashboardWidget |
| Order、Product（多服务共用） | 一次性原型实体 |
| 命名规范、ID 类型、时间格式 | 项目特有业务流程实体 |
| 通用 API 路径约定 | 项目专属接口 |

### 4.4 契约包引用规则

```yaml
project: ecommerce-api
imports:
  - pack: auth-pack
    version: ^1.2.0
  - pack: commerce-pack
    version: 2.0.1
extensions:
  - entity: User
    add_fields: [last_login_ip]
local_entities:
  - name: AdminAuditLog
    fields: [...]
```

**规则：**

- ✅ 允许：给共享实体新增字段、新增关联
- ❌ 禁止：修改共享实体已有字段名/类型、删除共享字段
- 修改共享实体核心定义 → 必须到契约库操作，走版本升级与发布流程

**版本语义：**

- `minor`（新增字段）：引用项目可选同步
- `breaking`（改类型、删字段）：通知所有引用项目，需人工确认后升级

## 五、用户模式与部署

### 5.1 本地个人 vs 云端组织

| 维度 | 本地个人模式 | 云端组织模式 |
|---|---|---|
| **用户** | 独立核心开发者 | 3–10 人核心开发团队 |
| **工作区** | 个人工作区 | 组织工作区 |
| **契约库** | 个人契约库 | 组织契约库 |
| **项目** | 个人多个项目 | 组织内多个项目/仓库 |
| **协作** | 无 | 成员、权限、WebSocket、表决 |
| **AI Key** | 本机加密 | 本机加密，不上云 |
| **主存储** | SQLite（Source of Truth） | 云端 MySQL + 本地 SQLite 缓存 |
| **离线** | 完全可用 | 可离线编辑，重连后同步 |
| **定价** | 免费（个人试用） | 按团队席位收费 |

### 5.2 升级路径

```
个人本地试用 → 个人契约库打磨 → 发布到组织契约库 → 项目迁移至云端 → 接入 CI 卡点
```

### 5.3 三种部署拓扑

| 拓扑 | 场景 | 组件 |
|---|---|---|
| **纯单机** | 个人试用、本地校验 | 桌面 App + SQLite |
| **局域网** | 3 人同办公室协作 | 桌面 App + 局域网 Sync Server |
| **云端组织** | 远程分布式团队 | 桌面 App + 云端 MySQL + WebSocket |

## 六、契约规范

### 6.1 规范选型

采用 **「JSON Schema 核心规范 + Prisma 关系映射概念」** 混合型标准：

| 优势 | 说明 |
|---|---|
| 表达能力 | 精确定义实体、DB 类型映射、主外键、校验规则 |
| Git 友好 | 纯文本，支持 diff/merge/blame |
| AI 亲和 | LLM 对 JSON Schema 具备强 Zero-shot 解析能力 |
| 生态互通 | 可导出 SQL、Prisma schema、OpenAPI、TS interface |

### 6.2 契约文件示例

```json
{
  "$schema": "https://synkord.dev/schema/contract-v1.json",
  "pack": "auth-pack",
  "version": "1.2.0",
  "conventions": {
    "id_type": "uuid",
    "naming": {
      "db": "snake_case",
      "api": "camelCase",
      "java": "PascalCase"
    },
    "timestamps": {
      "created_at": "datetime",
      "updated_at": "datetime"
    }
  },
  "entities": {
    "User": {
      "table": "users",
      "fields": {
        "id": { "type": "uuid", "primary": true },
        "email": { "type": "string", "maxLength": 255, "unique": true },
        "status": { "type": "enum", "values": ["active", "suspended"] }
      },
      "relations": {
        "roles": {
          "type": "many-to-many",
          "target": "Role",
          "through": "user_roles"
        }
      }
    }
  }
}
```

### 6.3 多端命名映射

契约统一定义命名规则，校验时自动检查：

```
契约字段: email
  → DB:        email        (snake_case)
  → API/TS:    email        (camelCase)
  → Java DTO:  email        (camelCase)
  → Go:        Email        (PascalCase export)
```

## 七、三层混合校验引擎

### 7.1 架构

```
代码产出 / PR 提交
    ↓
┌─────────────────────────────────┐
│ L1 极速基础过滤（Rust，毫秒级）    │  ← CI 默认
│ 正则 / JSONPath / 命名规则匹配     │
└──────────────┬──────────────────┘
               ↓  --deep 可选
┌─────────────────────────────────┐
│ L2 LLM 智能裁判（可选）           │  ← 重要 PR / 契约变更审阅
│ 跨语言语义级比对                  │
└──────────────┬──────────────────┘
               ↓  企业版
┌─────────────────────────────────┐
│ L3 开放插件（企业版）              │
│ 自研 AST 静态分析 Validator       │
└─────────────────────────────────┘
```

### 7.2 各层职责

| 层级 | 技术 | 拦截示例 | 默认场景 |
|---|---|---|---|
| **L1** | Rust 正则/JSONPath | 缺少 `email` 字段；DB 列名使用 camelCase | CI 每次 PR |
| **L2** | LLM Judge | DTO 未完整表达 Order 必填字段 | `synkord check --deep` |
| **L3** | AST 插件 | 企业内部 Java 注解规范 | 企业定制版 |

### 7.3 校验覆盖

| 产出物 | P0 | P1 |
|---|---|---|
| SQL 建表语句 | ✅ | ✅ |
| TS Interface / type | ✅ | ✅ |
| Java Entity / DTO | — | ✅ |
| Go struct | — | ✅ |
| Vue/React API 函数 | — | ✅ |
| OpenAPI spec | — | ✅ |

## 八、协同机制：契约变更表决

> 适用于云端组织模式。本地个人模式无此流程，直接发布即可。

### 8.1 四阶段状态机

```
阶段1          提交审查         阶段2          全票通过         阶段3          正式发布         阶段4
编辑待提交  →  提交待表决   →  待发布      →  已发布
(编辑锁)       (Diff推送)      (自动测试)     (全员同步)
    ↑              │
    └──── 驳回 ────┘
```

| 阶段 | 触发 | 系统动作 |
|---|---|---|
| **1 编辑待提交** | 成员开始修订 | WebSocket 广播编辑锁；其他人只读 |
| **2 提交待表决** | 修订人提交审查 | 生成变更 Diff 图谱推送全员 |
| **3 待发布** | 表决通过 | 锁定版本，执行自动格式校验 |
| **4 已发布** | 确认发布 | 新版入库，WebSocket 推送，AI 上下文重载 |
| **驳回** | 任一人驳回 | 退回阶段 1，附驳回理由 |

### 8.2 表决规则

| 规则 | 默认值 |
|---|---|
| 表决方式 | 全票通过（一票否决） |
| 可配置 | 支持切换为多数决（2/3） |
| 紧急发布 | Owner 可 override，强制发布（记录审计日志） |
| 表决超时 | 48 小时未响应视为弃权（P1） |

## 九、兜底与降级机制

| 级别 | 名称 | 动作 | 触发条件 |
|---|---|---|---|
| **Level 1** | 机器自治自愈 | 违规报告 + 契约打包 → 打回 AI 重写，最多静默 3 次 | AI 生成不合规 |
| **Level 2** | 局部采纳 | Diff 高亮冲突区域 → 开发者手动修改 → 合规后保存 | 3 次重试仍有瑕疵 |
| **Level 3** | 人工强制接管 | 冻结 Agent → 展示原始输出 →「强制采纳」或「废弃」 | AI 彻底失败或大型重构 |

所有 Level 3「绕过校验」操作记录操作人、时间、理由，写入审计日志。

## 十、Agent 与多模型接入

### 10.1 Agent 定位

Agent 是 **辅助能力**，不是产品主轴：

| 场景 | 说明 |
|---|---|
| AI 起草契约 | 从需求描述或现有源码反向生成 JSON Schema 契约 |
| AI 修复违规 | Level 1 自动重试的执行者 |
| AI 辅助评审 | 表决阶段，L2 Judge 辅助团队评估变更影响 |

### 10.2 多模型接入

**两条独立扩展线：**

| 类型 | 职责 | 示例 |
|---|---|---|
| **Model Provider** | 发 AI 请求 | OpenAI、Claude、豆包、通义、Ollama、自定义 URL |
| **Capability / Tool** | Agent 可用能力 | 文件读写、终端、Git、契约读写、校验、MCP、Skills |

**核心规则：**

- 用户 Key 本机 AES 加密，平台不托管、不中转
- 请求前自动注入有效契约 + Skills
- 同一项目不同成员可使用完全不同模型

### 10.3 Skills

| 类型 | 范围 | 内容 |
|---|---|---|
| 个人 Skills | 本地个人工作区 | 个人编码偏好 |
| 团队 Skills | 云端组织工作区 | 代码规范、技术栈模板、注释标准 |

Skills 与契约形成双重约束：底层数据结构 + 上层编码风格。

## 十一、产品界面

### 11.1 桌面 App 结构（核心开发主界面）

团队内所有核心开发均使用完整桌面 App：

```
Synkord 桌面 App
├── 工作区切换（个人本地 / 云端团队）
├── 📦 契约库（契约包管理、版本历史）
├── 📁 项目列表（引用关系、有效契约预览）
├── 📝 双栏架构工作台
│     ├── 左：JSON/YAML 契约编辑器（Monaco，自动补全）
│     └── 右：实体拓扑图 / ER 图（毫秒级联动）
├── 🔄 变更管理（Diff 图谱、表决状态、发布历史）
├── 🤖 AI 助手（辅助起草/修复，轻量侧边栏）
├── ✅ 校验报告（违规列表、行号定位）
└── ⚙️ 设置（AI 模型、Skills、CLI 配置）
```

### 11.2 CLI/CI 工程探针（嵌入开发流程）

CLI 与 CI 不是给「另一类人」用的，而是核心开发日常工程流程的一部分：

```bash
synkord init                              # 项目关联团队契约
synkord check ./src                       # 提交前本地 L1 校验
synkord check --deep ./src                # 深度 L1 + L2 校验
synkord report                            # 查看当前引用的契约版本
```

- **本地**：开发过程中随时 `synkord check`，配合 App 内校验
- **CI**：PR 时自动跑 `synkord check`，团队统一卡口

## 十二、CI/CD 工程探针

### 12.1 CLI 工具

```bash
npm install -g @synkord/cli

synkord init
synkord check ./src
synkord check --deep ./src
synkord check --contract auth-pack@1.2.0
```

### 12.2 GitHub Actions 集成

```yaml
name: Synkord Contract Check
on: [pull_request]
jobs:
  contract-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: synkord/check-action@v1
        with:
          contract-url: ${{ secrets.SYNKORD_CONTRACT_URL }}
          level: basic
          fail-on-error: true
```

### 12.3 团队工程闭环

```
核心开发团队在 Synkord 定契约并发布
  → 团队各成员在 App 内开发，AI 产出自动过校验
    → 代码库接入 synkord check CI 卡点
      → 任何成员（及 AI）产出不合规即标红
        → 团队付费扩展云端协同（表决、审计、多项目治理）
```

CLI 与 CI Action 随产品免费提供，作为核心开发工程流程的标准组件。

## 十三、技术架构

### 13.1 技术栈

| 层 | 选型 | 职责 |
|---|---|---|
| 壳 | Electron + Node.js | 跨平台桌面、IPC、系统权限 |
| UI | TypeScript + React + Radix UI + Monaco | 双栏工作台、Diff、设置 |
| 内核 | Rust（IPC sidecar） | 校验引擎、契约解析、模型调度、导出 |
| 本地存储 | SQLite (better-sqlite3) | 个人全量；组织本地缓存 |
| 云端存储 | MySQL + 对象存储 | 组织、契约库、成员、版本 |
| 协同 | WebSocket + REST | 编辑锁、表决、契约推送 |
| 资产 | Git（可选） | 迁移脚本、OpenAPI 导出 |

### 13.2 进程架构

```
React 渲染进程
    ↓ Electron IPC
Node.js 主进程
    ↓ IPC / stdio
Rust 内核
    ├── 契约解析器（resolve → 有效契约）
    ├── L1 校验引擎
    ├── L2 Judge 调度器
    ├── Model Provider 适配器
    ├── 导出器（SQL / TS / Java / OpenAPI）
    └── 沙箱执行器
```

### 13.3 存储职责

| 数据 | 本地个人 | 云端组织 |
|---|---|---|
| AI Key | 本机加密，不上云 | 同左 |
| 契约库 | SQLite 主存储 | MySQL 主存储 + SQLite 缓存 |
| 项目元数据 | SQLite | MySQL + 本地缓存 |
| 审计日志 | 本地 | 云端（付费功能） |
| 离线真相源 | 本地 SQLite | 重连后以云端为准 |

## 十六、MVP 分期

### Phase 1 — P0 垂直切片（8–12 周）

**目标：跑通「定契约 → 校验 → CI 拦截」核心价值闭环。**

| 模块 | 交付范围 |
|---|---|
| 桌面 App | Electron + React + Monaco 双栏工作台 |
| 工作区 | 本地个人工作区 + 个人契约库 + 多项目引用 |
| 契约格式 | JSON Schema 子集（实体、字段、关联、命名约定） |
| ER 预览 | 右侧实时拓扑图 |
| 导出 | MySQL SQL + TS Interface |
| 校验 L1 | Rust 引擎（TS + SQL） |
| 校验 L2 | `synkord check --deep`（1 个云端模型） |
| 模型接入 | OpenAI + Ollama |
| Agent | 轻量辅助：AI 起草契约、自动修复违规 |
| CLI | `synkord check` + `synkord init` |
| CI | GitHub Actions 模板 |
| 存储 | SQLite |
| 降级 | Level 1 自动重试 + Level 3 人工接管 |
| 发布 | 手动发布版本（无表决流程） |

**P0 验证标准：**

> 两个本地项目共用 `auth-pack`；成员产出的 TS 代码缺少 `User.email` → App 内校验标红 → `synkord check` 本地复现 → CI 拦截 PR。

### Phase 2 — 组织协同（+8 周）

- 云端组织工作区 + 组织契约库
- 成员邀请 + 权限模型
- WebSocket 编辑锁 + 四阶段表决
- 契约版本管理 + 变更影响分析
- 个人契约库 → 组织迁移
- 局域网部署模式

### Phase 3 — 全栈与增强（+8 周）

- Java / Go / Vue / React 校验与导出
- 团队 Skills
- Level 2 局部采纳（Diff 高亮手动修复）
- VS Code 插件
- 审计日志

### Phase 4 — 企业版

- L3 AST 插件接口
- 可配置表决规则
- 自动化巡检（Automations）
- 多组织管理
- 私有化部署

## 十七、商业化

### 17.1 GTM 策略

聚焦 **3–10 人核心开发团队**，打造为契约协同、架构防腐、AI 合规开发的一体化桌面工具。

### 17.2 增长漏斗

```
个人核心开发者免费试用（本地单机）
  → 认可契约库 + 校验价值
    → 邀请 2–9 名队友组成云端团队
      → 团队接入 CI 卡点，统一工程标准
        → 团队付费（云端协同、表决、审计）
```

### 17.3 定价

| 层级 | 价格 | 包含 |
|---|---|---|
| **Personal** | 免费 | 本地工作区、个人契约库、桌面 App、L1 校验、CLI/CI |
| **Team** | ¥99/人/月（3 人起） | 团队契约库、WebSocket 表决、版本管理、L2 Judge |
| **Enterprise** | 议价 | AST 插件、审计日志、私有化部署、SLA |

所有层级均包含完整桌面 App 与 CLI/CI，按团队规模与协同能力分级收费。
